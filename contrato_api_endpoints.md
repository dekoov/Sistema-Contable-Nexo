# Contrato API REST — Migración Monolito → Distribuido

Basado en `source_code_legado.xml` (repomix). Módulos cubiertos: `comun`, `facturacion`, `inventario` (según OBJETIVO). Módulo `cxc` existe en código pero fuera de alcance declarado — omitido.

Todo endpoint responde wrapper `ApiResponse<T>` (ver doc anterior AuthResource):
```json
{ "status": "success|error", "message": "string", "data": {} }
```

**Riesgo transversal Oracle→Postgres:** todo `Negocio*.obtenerSiguienteId()` usa `MAX+1` manual (no SEQUENCE/IDENTITY). Bajo concurrencia real (2+ nodos app) esto colisiona PK. Migrar a `GENERATED BY DEFAULT AS IDENTITY` en Postgres y quitar estos métodos del contrato REST — el backend ya no calcula ID, lo genera la BD. Marco esto en cada entidad abajo.

**Riesgo tipos:** `idArticulo`, `idComprobante`, `idTipoMovimiento`, `idComprobanteDet` son `BigDecimal`/`BigInteger` en inventario, pero `Integer` en `FacturaDetalle.idArticulo` — conversión manual ya existe en `NegocioComprobante.registrarEgresoPorVenta` (`BigDecimal.valueOf(detFactura.getIdArticulo())`). Mantener ese mapeo al construir DTOs.

Origen: `NegocioUsuario.java`

| Método | Path | Origen negocio | Rol mínimo |
|---|---|---|---|
| POST | `/auth/login` | `login()` | público (ya documentado) |
| GET | `/usuarios` | `listarTodos()` | ADMIN |
| GET | `/usuarios/{id}` | `em.find` (no existe método `buscar`, agregar) | ADMIN |
| POST | `/usuarios` | `insertar()` | ADMIN |
| PUT | `/usuarios/{id}` | `modificar()` | ADMIN |
| DELETE | `/usuarios/{id}` | `eliminar()` | ADMIN |

**DTO `UsuarioDTO`** (data):
```json
{ "idUsuario": 1, "username": "jdoe", "rol": "USER", "fechaCreacion": "2026-07-01" }
```
`contrasena` nunca sale en response — ya no aplica tras migrar a OAuth2 (campo queda legado, no se expone).

**Riesgo:** `insertar`/`modificar` no llaman `obtenerSiguienteId` internamente — la pantalla JavaFX lo hacía antes de construir el objeto. En REST, el backend debe asignarlo (o Postgres IDENTITY, ver arriba). No inventar wrapper: usar patrón `ApiResponse.error()` si `idUsuario` viene nulo en insert.

## Desacople JMS — Factura → Inventario

Reemplaza llamada síncrona `NegocioFactura.insertar()` → `NegocioComprobante.registrarEgresoPorVenta()`. Cumple restricción demo "evidencia cola: dato visible → importar → cola vacía".

**Cola:** `jms/facturaCreadaQueue` (Payara JMS resource, tipo Queue)

**Mensaje (JSON, no serialización Java nativa — evita acoplar clases entre .war si se separan módulos a futuro):**
```json
{
  "idFactura": 10,
  "numeroFactura": "F-0010",
  "fecha": "2026-07-10",
  "detalles": [
    { "idArticulo": 5, "cantidad": 2, "precio": 75.25 }
  ]
}
```

**Productor** — dentro de `AuthResource`-style JAX-RS resource de facturación, tras `tx.commit()` del insert de cabecera+detalle:

```java
// backend .war modificado — facturacion/negocio/NegocioFactura.java (reemplaza bloque actual)
public int insertar(FacturaCabecera factura) {
    EntityManagerFactory emf = null;
    EntityManager em = null;
    EntityTransaction tx = null;
    try {
        emf = Persistence.createEntityManagerFactory(PU);
        em  = emf.createEntityManager();
        tx  = em.getTransaction();
        tx.begin();

        em.persist(factura);
        for (FacturaDetalle det : factura.getDetalles()) {
            det.setFactura(factura);
            em.persist(det);
        }

        tx.commit();

        publicarEventoFacturaCreada(factura); // reemplaza llamada directa a NegocioComprobante

        return 1;
    } catch (Exception e) {
        if (tx != null && tx.isActive()) tx.rollback();
        e.printStackTrace();
        return -1;
    } finally {
        if (em  != null) em.close();
        if (emf != null) emf.close();
    }
}

private void publicarEventoFacturaCreada(FacturaCabecera factura) {
    try {
        jakarta.jms.ConnectionFactory cf =
            jakarta.enterprise.inject.spi.CDI.current().select(jakarta.jms.ConnectionFactory.class).get();
        jakarta.jms.Queue queue =
            jakarta.enterprise.inject.spi.CDI.current().select(jakarta.jms.Queue.class).get();
        try (jakarta.jms.JMSContext ctx = cf.createContext()) {
            String jsonMsg = construirJsonFactura(factura); // serializar con Jsonb/JSON-B
            ctx.createProducer().send(queue, jsonMsg);
        }
    } catch (Exception e) {
        System.out.println("Advertencia: factura guardada, evento JMS no pudo publicarse: " + e.getMessage());
    }
}
```

```java
    @ActivationConfigProperty(propertyName = "destinationType", propertyValue = "jakarta.jms.Queue")
})
public class EgresoPorVentaListener implements MessageListener {

    @Override
    public void onMessage(Message message) {
        try {
            String json = ((TextMessage) message).getText();
            // parsear JSON -> reconstruir datos mínimos, NO reutilizar entidad FacturaCabecera completa

```

**Trade-off:** async rompe consistencia inmediata (factura visible antes que descuento de stock aplique). Para demo, aceptable — el video muestra justamente ese delay como evidencia de cola. En prod real se agregaría estado `PENDIENTE_INVENTARIO` en factura, fuera de alcance académico.
